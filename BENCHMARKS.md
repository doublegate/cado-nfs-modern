# Benchmarks

Reference performance of **CADO-NFS 2.3.1-modern** factoring balanced
(RSA-like) semiprimes on a single desktop. These numbers characterize this
build on this class of hardware; they are not a comparison against other NFS
implementations.

## Test machine

| Component | Detail |
|-----------|--------|
| CPU | Intel Core i9-10850K — 10 cores / 20 threads, 3.6 GHz base / 5.2 GHz boost (Comet Lake) |
| RAM | 64 GiB DDR4 |
| OS / kernel | CachyOS, Linux 7.0.10 |
| Compiler | GCC 16.1.1, `-O2 -fcommon` |
| Libraries | GMP 6.3.0, hwloc 2.13.0 |
| Orchestration | Python 3.14.5 |
| CADO-NFS | 2.3.1-modern (this fork) |

## Methodology

- **Inputs:** balanced semiprimes `n = p·q` with `p`, `q` of equal digit length
  (the hardest case for NFS). Generated by a Miller-Rabin routine seeded with
  `20260604` for reproducibility; the exact inputs are listed below.
- **Command:** `./cado-nfs.py <n> server.ssl=no -t 20` — all 20 logical threads;
  TLS transport disabled to isolate compute (it adds <1 s of cert setup and is
  irrelevant to the result either way).
- **Timing:** wall-clock measured around the process; CPU and elapsed times as
  reported by CADO-NFS (`Total cpu/elapsed time for entire factorization`).
  `parallel` = total CPU ÷ elapsed (effective speedup across the whole run).
- **Runs:** one run per size. NFS timing has inherent ~±15-20 % variance, mostly
  from randomized polynomial selection; treat these as representative, not exact.
- Every factorization was verified: the reported factors multiply back to `n`
  and each is prime.

### Per-phase CPU time (seconds)

CADO-reported CPU per sub-program. At c90 these account for ~99.8 % of the
headline CPU (sieving + filtering + linalg + sqrt = 1938 of 1943); at very small
sizes the breakdown is noisy due to measurement granularity and un-itemized
setup. Polynomial selection is reported separately as wall-time (right column)
rather than CPU, so it is not added into the total.

| Digits | Lattice sieving | Filtering | Linear algebra | Square root | (Polyselect, wall) |
|-------:|----------------:|----------:|---------------:|------------:|-------------------:|
| 60 |   39.4 |  20.9 |   3.7 |  0.7 |  3.3 s |
| 70 |   82.9 |  45.0 |   5.2 |  2.1 |  ~6 s |
| 80 |  342.2 |  44.0 |  39.1 |  5.1 | 14.2 s |
| 90 | 1409.7 | 130.6 | 366.2 | 31.7 | 64.4 s |

("Filtering" sums dup1/dup2/purge/merge/replay.)

## Observations

- **Sieving dominates** (61-73 % of all CPU) and is the embarrassingly-parallel
  phase. This is why **parallel efficiency climbs with size** — 1.9× at c60
  (where fixed sequential overhead swamps the tiny sieve) up to 11.1× at c90.
  During the sieving phase alone the machine runs close to all-cores-busy; the
  whole-run average is pulled down by the sequential linear-algebra and Python
  orchestration around it.
- **Linear algebra grows the fastest** of any phase: 3.7 → 5.2 → 39.1 → 366 CPU-s
  from c60 to c90 (~100×, vs ~36× for sieving over the same span). Block
  Wiedemann scales worse than sieving and becomes the clear second bottleneck as
  numbers grow — the classic NFS trade-off. (Its wall-clock is mis-reported as
  ~0 by this CADO version; the CPU figure is the meaningful one.)
- **Growth rate.** Wall-time roughly doubles per +10 digits in this range
  (c70→c80 2.1×, c80→c90 2.4×); CPU work grows faster (3.5-4.6×). This matches
  the sub-exponential `L(1/3)` complexity of NFS — the per-decade factor itself
  slowly shrinks at larger sizes.

## Rough projections (this machine)

Extrapolated from the measured CPU growth (~3.5-4× per +10 digits) and rising
parallel efficiency. **Order-of-magnitude only** — real numbers depend on
parameter tuning, memory bandwidth, and luck in polynomial selection.

| Digits | Projected wall time | Practicality |
|-------:|--------------------:|--------------|
| ≤ 75 | < 1 min | interactive |
| 80-95 | 1-5 min | coffee break |
| 100 | ~10-15 min | single sitting |
| 110 | ~45-90 min | long session |
| 120 | several hours | overnight |
| ≥ 130 | a day or more | wants a cluster / distributed mode |

For this 10-core desktop, **up to ~c105-c110 is comfortable in one session**;
beyond ~c120, use multiple machines (CADO-NFS distributed mode) or expect
multi-day runs. RAM (64 GiB) is not the limit in this range — sieving and the
matrix stay well within it until well past c120.
